Algorithmic ideas¶

  • Complexity: quantifies the computation needs
  • How to multiply two numbers a and b: a * 111
    • $\underbrace{a + a + ... + a}_{111~times}$
  • Multiplication strategy learned at school: $a \cdot 100 + a \cdot 10 + a \cdot 1$
  • In binary: $$4 \cdot 3 = 100 \cdot 11 = 100 << 1 + 100 << 0 = 1000 + 100 = 1100 = 2^3 + 2^2 = 8 + 4 = 12$$

  • The efficiency of an algorithm is much more important than the technology/harware used to execute it.

    • Exemple of series $\sum_k^N k = N(N+1)/2$
    • The complexity of the naive algorithm to compute a N-body problem is $O(N^2)$

Code optimization¶

Compiler optimization option¶

  • In order to ask the compiler, one should ask for compilation with optimization.
gcc -c file.cc -O3
  • while for debugging the compiler should be required to attach debugging symbols
gcc -c file.cc -g

In CLion, you can get this option with the Configuration menu in the CMake settings.

Let's try this:

int foo(const int &i) { return i + 18; }

int main() {

  unsigned long res = 0;
  for (long i = 1; i < 1000000000; ++i) {
    res += i + foo(i);
  }
  std::cout << "Hello, World! " << res << std::endl;
  return 0;
}

GNU profiler¶

There are several tools to profile a code which aims at helping the optimization process.

  • The GNU profiler: gprof. You need to compile the sources with an option to GCC.
g++ -c -pg file.cc -g
g++ -pg -o program_name file.o

To use it in CLion, you should add the following in your CMakeLists.txt

set(CMAKE_CXX_FLAGS "-pg")
set(SOURCE_FILES main.cpp test.cpp)

Then you launch your program normally and a file gmon.out will be produced which can be analysed with the program gprof.

gprof PROGRAM_NAME gmon.out

For a line by line information you can use:

gprof -l PROGRAM_NAME gmon.out

Valgrind & Cachegrind¶

  • Valgrind: running your program through valgrind can allow to profile the call graph
valgrind --tool=callgrind PROGRAM_NAME arguments
  • A file is created: callgrind.out.pid
  • It can be analyzed with kcachegrind (only under Unix-like OS, sorry).

Perf Tool¶

  • Instruments CPU performance counters
  • Lightweight profiling
  • Included in the Linux kernel, under tools/perf
  • Frequently updated and enhanced 
perf record -g program

Virtual table and calls¶

  • Virtual table (virtual method table, virtual function table, virtual call table, dispatch table, or vtable, VMT or Vtable)
  • Mechanism supporting dynamic dispatch (or run-time method binding)
  • Compilers add a hidden pointer to the virtual method to call table
  • Concretely a VMT is an array of pointers to (virtual) functions

Consider the following:

struct ComputeInterface {
  virtual double compute() = 0;
};

struct A : public ComputeInterface {
  virtual double compute() { return M_PI; };
};

struct B : public ComputeInterface {
  virtual double compute() { return M_E; };
};

If we manipulate objects with the mother class type e.g.:

double performCompute(ComputeInterface &compute) { return compute.compute(); }

int main() {
  A a;
  performCompute(a);
  B b;
  performCompute(b);
}

With the virtual table:

double computeMethodA(ComputeInterface * _this){return M_PI;};
double computeMethodB(ComputeInterface * _this){return exp(1.);};

struct ComputeInterface { FunctionPointer computeMethodPtr; };

double performCompute(ComputeInterface * compute){
  return compute->computeMethodPtr(compute);
}

With the following call

void main(){
  ComputeInterface a;
  a.computeMethodPtr = computeMethodA; performCompute(&a); 
  ComputeInterface b;
  b.computeMethodPtr = computeMethodB; performCompute(&b);
}

The most dramatic effect of virtual function calls is that it prevents (in most cases) the possibility to inline.

This should be avoided in the critical loops

On the critical loops¶

Scientific codes have large loops $\Rightarrow$ The critical loops

  for (unsigned int a = 0; a < nb_atoms; ++a) {
    // do something for every atoms...
  }
  • When possible, decisions should be avoided in critical loops
  for (unsigned int a = 0; a < nb_atoms; ++a) {
    if (a % 2 == 0) {
      // do something ...
    } else {
      // do something else...
    }
    // generate an execution flow divergence
  }
  • Virtual calls should also be avoided in critical loops

About memory localization¶

  • The processor reads memory by chunks: pre-fetch of information in memory cache
  • Contiguous data access optimizes cache fill and avoids cache misses
  • In general, ordered memory accesses are more efficient than random memory accesses

Variadic template meta programming¶

template <typename T> T adder(T v) { return v; }

template <typename T, typename... Args>
// template ...Args is a variadic template
T adder(T first, Args... args) {
  return first + adder(args...);
}

Using it simply

  std::cout << adder(1, 2, 3, 4);
  std::cout << adder(1., 2, 3., 4);
  std::cout << adder(std::string("a"), std::string("b"));

About Functors and Lambdas¶

struct MyFunctor {
  int operator()() { return 2; }
};

int main() {
  auto f = MyFunctor();

}

struct MyFunctor {
  inline int operator()() { return a; }
  int a;
};

int main() {
  auto f = MyFunctor();
  int a = 2;
  f.a = a;

  for (int i = 0; i < 100000; ++i) {
    auto res = f();
  }
}

struct MyFunctor {
  inline int operator()() { return *a; }
  int *a;
};

template <typename T> void loop(T f) {
  for (int i = 0; i < 100000; ++i) {
    auto res = f();
  }
}

int main() {
  auto f = MyFunctor();
  int a = 2;
  f.a = &a;
  loop(f);
}
